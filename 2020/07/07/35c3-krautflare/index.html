漏洞分析

Patch

对于revert-bugfix-880207.patch

在typer.cc文件中，可以发现Math.expm1()返回的type为Union(PlainNumber, NaN)。

d8> Object.is(0, -0)
false

我们可以发现，在v8中，0和-0是不同的。其中-0的type为HEAP_NUMBER_TYPE，代表着浮点数。
而PlainNumber则代表除了-0以外的所有浮点数。
所以typer在处理Math.expm1的时候没有考虑到传入-0的情况。

Trigger

由前面的代码可知，typer在处理Math.expm1，返回为PlainNumber和NaN。所以Object.is(Math.expm1(x), -0)的返回值应该一直为false。我们通过以下代码进行验证。

foo(0)  : false
foo(-0) : true

得到的返回值显然和预料中的不一样，通过Turbolizer观察一下。

在simplified lowering中发现Math.expm1被替换为Float64Expm1，为了触发bug，我们需要添加foo("0")接受String类型参数，以此触发更底层的Call。

OOB

在上面Trigger中，我们能够得到固定的返回值false，再将其转化为数字，Turbofan会将其认为0，在simplified lowering阶段会去掉CheckBounds。但实际存在1的情况，由此触发OOB。
我们在v8中运行以下代码：

我们得到的结果都是oobArray[0]，说明idx一直为0。

在Typer阶段中，idx被替换为了0。
所以我们要找一种在simplified lowering都不会影响到BUG触发的方法，可以考虑simplified lowering的前一个阶段escape analysis。
我们可以把值放在非逃逸对象中以避免Typer阶段idx的替换。

foo(0)  : 1.1
foo(0)  : 1.1
foo(-0) : -1.1885946300594787e+148

在escape analysis阶段之前，tmp.escape以及tmp的类型都是不知道的，escape analysis阶段将tmp.escape替换为NumberConstant[-0]，接着在simplified lowering中的typer阶段，tmp.escape的typer被收集，由于Turbofan认为idx肯定为0，所以去除了CheckBounds，成功实现了OOB。
